# 更新日志 v3.9

日期：2026-03-17

## 版本概述

v3.9 实现"核心引擎与外壳架构"（Core & Shell Architecture），将 v3.8 引入的完整认证体系从核心工具逻辑中彻底解耦。通过 `ENABLE_AUTH` 环境变量，系统可在**纯工具模式**（无需登录）和**商业模式**（完整认证）之间一键切换，同一套代码库服务两种场景。

---

## 一、后端 — 认证抽象层

### 1) auth_deps.py（核心变更）

- 新增 `app/core/auth_deps.py`，作为认证解耦的**唯一接口层**。
- 定义 `CurrentUser` frozen dataclass（`id: str` / `username` / `role` / `email`），替代原先的 `User` ORM 对象，所有 API 端点统一使用此类型。
- 定义 `ANONYMOUS_USER` 哨兵（`id="anonymous"`, `role="admin"`）。
- 导出 `get_current_user` / `get_current_admin` — FastAPI `Depends()` 可调用对象：
  - `ENABLE_AUTH=true`：委托 `fastapi_users_config` 中的 `current_active_user` / `current_admin_user` 解析 JWT Cookie，将 `User` 转为 `CurrentUser`。
  - `ENABLE_AUTH=false`：直接返回 `ANONYMOUS_USER`，**不导入 fastapi-users**。

### 2) 配置扩展

- `app/core/config.py` 新增 `ENABLE_AUTH: bool` 字段（默认 `true`，从 `ENABLE_AUTH` 环境变量读取）。

### 3) 全部 API 端点迁移

所有 API 端点从 `from app.db.fastapi_users_config import current_active_user` 迁移到 `from app.core.auth_deps import CurrentUser, get_current_user`：

| 文件 | 变更 |
|------|------|
| `api/labels.py` | `User` → `CurrentUser`，`user.id` → `current_user.id`，`user_id=` → `owner_id=` |
| `api/batches.py` | 同上 |
| `api/templates.py` | 同上 |
| `api/system.py` | `current_admin_user` → `get_current_admin`，`User` → `CurrentUser` |
| `api/helpers.py` | `get_owned_record_or_404(owner_id: str)` 参数类型改为字符串；新增 `require_admin()` 委托 `get_current_admin` |

### 4) 条件路由注册

- `api/router.py`：auth 路由仅在 `settings.ENABLE_AUTH=true` 时 import 并注册。
- `app/main.py` lifespan：`ENABLE_AUTH=true` 时才导入并调用 `seed_default_users()`。

## 二、数据库 — 弱关联模型

### 5) models.py 重构

- `User` 类改为**条件定义**：`ENABLE_AUTH=true` 时正常定义（继承 `SQLAlchemyBaseUserTable`）；`false` 时 `User = None`，users 表不注册到 metadata、不创建。
- 三张业务表（`LabelHistory` / `LabelBatch` / `LabelTemplate`）的 `user_id: int`（FK → `users.id`）全部替换为 `owner_id: str`（`String(128)`，default `"anonymous"`），**不建 FK**。
- 索引更新：`ix_lb_owner_id_desc`、`ix_lh_owner_created`、`ix_lh_owner_id_desc`。

### 6) Alembic 迁移 0007

- 新增 `0007_decouple_auth_owner_id.py`，数据安全迁移：
  - **升级**：添加 `owner_id` 列 → 将已有 `user_id` 整数值 cast 为字符串写入 → 删除旧 FK / 索引 / `user_id` 列。
  - **降级**：将数字型 `owner_id` 转回 `user_id` 整数 → 删除工具模式产生的纯匿名数据行 → 恢复 FK。
  - 使用 `sa.inspect(bind)` 做幂等检查，避免重复执行时报错。

### 7) Schema 更新

- `schemas/label.py`、`batch.py`、`template.py`：所有响应模型中 `user_id: int` → `owner_id: str`。
- `LegacyLoginResponse.user_id` 类型改为 `str`。

## 三、前端 — 认证可选化

### 8) proxy.ts（Next.js 16 Proxy）

- 新增 `frontend/proxy.ts`（Next.js 16 的 `middleware.ts` → `proxy.ts` 正式迁移）。
- `NEXT_PUBLIC_ENABLE_AUTH !== "true"` 时，将 `/login`、`/register`、`/forgot-password`、`/reset-password`、`/verify-email` 重定向到 `/`。

### 9) auth.ts 改造

- `ENABLE_AUTH=false` 时：
  - `authUserCache` 预设 `MOCK_USER`（`user_id: "anonymous"`, `role: "admin"`），消除首次渲染闪烁。
  - `getAuthUser()` 直接返回 `MOCK_USER`。
  - `clearAuthUser()` / `initSession()` / `subscribeAuthUser()` 短路。

### 10) 其他前端适配

| 文件 | 变更 |

| `components/shared/Navbar.tsx` | 用户下拉菜单仅在 `ENABLE_AUTH && authUser` 时显示 |

## 六、架构决策要点

| 决策 | 选择 | 理由 |
|------|------|------|
| 认证抽象方式 | `CurrentUser` dataclass + 条件 Depends | API 层无需 `if` 分支，依赖注入自动切换 |
| 数据归属字段 | `owner_id: str`（无 FK） | 工具模式无 users 表，不能建 FK；string 类型兼容未来多种 ID 格式 |
| `ENABLE_AUTH` 默认值 | `true` | 商业模式为主，纯工具模式需显式 opt-in |
| User 模型加载 | 模块级 `if settings.ENABLE_AUTH:` | 避免在工具模式导入 fastapi-users 和注册 users 表到 metadata |
| 前端 auth 页面处理 | Next.js Proxy 重定向 | 比条件渲染更可靠，URL 级别阻断 |
| 前端 auth 状态初始化 | eagerly set `MOCK_USER` | 消除 SSR 水合不匹配和"checking auth"闪烁 |

## 七、迁移指南

### 从 v3.8 升级

1. 设置环境变量：`.env` 中添加 `ENABLE_AUTH=true`（保持原有行为）和 `NEXT_PUBLIC_ENABLE_AUTH=true`。
2. 运行数据库迁移：`alembic upgrade head`（执行 0007，自动将 `user_id` → `owner_id`）。
3. 重启前后端。

### 切换到纯工具模式

1. `.env` 中设置 `ENABLE_AUTH=false` 和 `NEXT_PUBLIC_ENABLE_AUTH=false`。
2. 重启前后端。无需数据库迁移——业务表的 `owner_id` 默认值 `"anonymous"` 会自动生效。
