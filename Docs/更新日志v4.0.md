# 更新日志 v4.0

## 概述

v4.0 专注于三个方向：**编辑器体验**、**预览一致性** 和 **生产部署加固**。

---

## 1. 编辑器体验重构：displayScale 坐标系

### 问题

旧版用 CSS `transform: scale(zoom)` 整体缩放画布。副作用：选中框从 2px 变成 `2×zoom px`，马奎选区框也随之放大，体验不专业。

### 解法

引入 `displayScale` 常量（默认 2，滑块可调）替代 CSS transform。  
所有设计坐标 **直接乘以 `displayScale` 写入 DOM**，无任何 CSS 缩放。

```
// Canvas.tsx 核心变化（简化示意）
<Rnd
  size={{ width: el.w * ds, height: el.h * ds }}
  position={{ x: el.x * ds, y: el.y * ds }}
  scale={1}   // ← 不再依赖 react-rnd 的 scale 修正
  onDragStop={(_e, d) => updateElement(id, { x: d.x / ds, y: d.y / ds })}
/>
```

**效果**：

| 指标 | 旧版 | v4.0 |
|------|------|------|
| 选中框宽度 | `2 × zoom px` | 固定 **1px**（物理像素） |
| 编辑舒适度 | 60×30mm 标签在屏幕只有几厘米宽 | 默认 2× 渲染，标签明显放大 |
| 缩放滑块单位 | `%`（100%=1×） | `×`（倍数，0.5～8） |
| Store 坐标 | 随 zoom 变化感知不稳定 | 永远是设计单位，与屏幕无关 |

### 马奎框选（react-selecto）

- `dragCondition` 确保框选只在画布空白处发起，不误触元素
- `data-el-id` 属性携带元素 id，选后直接写入 `selectedIds`
- 选中框始终是屏幕原生 1px `box-shadow`

## 2. 预览与导出一致性

### 问题

单次生成预览弹窗"默认布局"使用 `PreviewTemplateCanvas`（硬编码 HTML），与导出 SVG 的 `renderCustomSvg()` 路径完全不同。用户看到的预览和拿到的文件不一样。

### 解法

所有预览路径统一走 `renderCustomSvg()`：

| 入口 | 旧版渲染引擎 | v4.0 渲染引擎 |
|------|------------|--------------|
| 单次生成，未选模板 | `PreviewTemplateCanvas`（HTML） | `renderCustomSvg(sys-compact)` |
| 单次生成，选了模板 | `renderCustomSvg(模板)` | `renderCustomSvg(模板)` ✓ |
| 批量导出 | `renderCustomSvg(模板)` | `renderCustomSvg(模板)` ✓ |
| 编辑器 SVG 导出 | `renderCanvasToSvg(canvas)` | 同左 ✓ |

`PreviewTemplateCanvas` 从 `PreviewDialog` 中移除，预览容器改为 `flex justify-center` 居中显示。

### SVG 文本垂直对齐修复

旧版 SVG 文本 `y = el.y + el.fontSize`（基线紧靠顶部），与编辑器 `flex items-center` 居中对齐不一致。

修复：

```xml
<!-- 旧 -->
<text y="25" ...>LOT001</text>

<!-- v4.0 -->
<text y="22.5" dominant-baseline="central" ...>LOT001</text>
```

`dominant-baseline="central"` 将文本视觉中心对准 `y` 坐标，与编辑器完全一致。

### SVG 导出尺寸修复

旧版导出 `width="227"height="113"`（设计单位原始值），浏览器按像素打开只有指甲大。

v4.0 导出 `width="${W*3}" height="${H*3}" viewBox="0 0 ${W} ${H}"`，以 3× 物理尺寸打开，视觉还原与编辑器 `displayScale=3` 等效。

## 3. 生产部署加固

### Redis 密码认证

旧版 Redis 无任何认证，Docker 内网任何进程均可访问。

v4.0 `docker-compose.prod.yml`：

```yaml
redis:
  command: ["redis-server", "--requirepass", "${REDIS_PASSWORD}"]
```

连接串同步更新：`redis://:${REDIS_PASSWORD}@redis:6379`。

### 内存限制（2 核 2 GB 服务器适配）

各服务均设 `deploy.resources.limits.memory`，防止单服务 OOM 拖垮整机：

| 服务 | 内存上限 | 说明 |
|------|---------|------|
| postgres | 256 MB | |
| backend | 400 MB | 2 worker × ~150 MB |
| redis | 64 MB | |
| frontend | 256 MB | Next.js 运行时 |
| nginx | 32 MB | |
| **合计** | **~1 GB** | 留 1 GB 给 OS 和构建峰值 |

### Gunicorn worker 默认值调整

`WORKERS` 默认从 4 → **2**，适配 2 核服务器。4 worker 在 2 GB 环境下可能导致 OOM。

### Next.js 构建内存限制

```dockerfile
ENV NODE_OPTIONS=--max-old-space-size=512
```

Next.js 构建默认吃 1.5 GB+，此行将堆上限钉死 512 MB，构建在 2 GB 服务器上不再崩溃（速度略降）。

### Frontend 容器健康检查

旧版 nginx 在 frontend 容器仅 `service_started` 时就开始路由，导致冷启动短暂 502。

v4.0 添加健康检查，nginx 等待 `service_healthy`：

```yaml
healthcheck:
  test: ["CMD-SHELL", "node -e \"require('http').get('http://localhost:3000/', ...)\""]
  start_period: 60s
```

### COOKIE_SECURE 默认值修正

生产 compose 将 `COOKIE_SECURE` 默认值从 `true` 改回 **`false`**，适配 HTTP 初次部署。文档和 `.env.example` 均已注明：配好 HTTPS 后再手动设为 `true`。
