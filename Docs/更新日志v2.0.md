# 更新日志 v2.0 — POC 升级至生产级架构

日期：2026-03-15

---

## 版本目标

将 v1.x 阶段完成功能验证的 POC 代码，全面升级为可支撑多用户、高并发访问的生产可用状态。

本次升级聚焦三个已确认的瓶颈问题：

| # | 问题描述 | 根因 | 本次解法 |
|---|----------|------|----------|
| 1 | 历史记录首次加载慢/超时 | 无复合索引、offset 全表扫描、列表携带两列大 BLOB 字段 | 复合索引 + TanStack Query 缓存 + 删除 BLOB 列 |
| 2 | 后端多请求并发时间接超时 | SQLite 写锁、同步 SQLAlchemy 阻塞 uvicorn 事件循环、单进程 | 切换 PostgreSQL + 全链路 async SQLAlchemy + Gunicorn 多 worker |
| 3 | 条码预览延迟高（每次预览都要请求后端渲染） | 条码生成依赖后端 Ghostscript/treepoem，Preview 接口同步耗时 | 前端 bwip-js 全量客户端渲染，仅导出时向后端保存元数据 |

---

## 架构决策记录（ADR）

升级前，以下关键决策经过评估选型：

### ADR-1：数据库迁移至 PostgreSQL
- **选项**：SQLite + WAL / PostgreSQL / MySQL
- **决策**：PostgreSQL 16（asyncpg 驱动）
- **理由**：
  - asyncpg 是 Python 生态中性能最高的 async PG 驱动
  - PostgreSQL 对多 worker 写并发无 SQLite 行级写锁问题
  - 与 SQLAlchemy 2.0 async engine 原生集成

### ADR-2：并发模型
- **选项**：增加 uvicorn workers / 切换 asyncpg / 引入任务队列
- **决策**：全链路 async SQLAlchemy + Gunicorn + UvicornWorker（4 workers 默认）
- **理由**：
  - 条码渲染已从请求路径移除，后端业务变轻量（纯 DB 操作）
  - Gunicorn 多进程利用多核，UvicornWorker 保留 async 能力
  - 不需要 Celery/Redis 等额外组件

### ADR-3：前端缓存
- **决策**：TanStack Query v5（`@tanstack/react-query`）
- **理由**：
  - staleTime=30s 消除重复请求
  - keepPreviousData 消除翻页时的"数据闪烁"
  - invalidateQueries 精确让删除后的历史列表失效
  - 业界标准，文档完善，SSR 支持良好

### ADR-4：条码渲染与保存时机
- **决策**：预览阶段完全在浏览器端由 bwip-js 渲染；仅在用户点击导出（PNG/SVG/PDF）时调用 `POST /generate` 向后端保存元数据
- **影响**：
  - `/labels/preview` 和 `/labels/preview-svg` 接口**已删除**（破坏性变更）
  - 后端不再存储 `datamatrix_base64` / `gs1_128_base64` BLOB 列
  - 历史预览通过前端从元数据重渲染

### ADR-5：GS1 逻辑前后端同构
- **决策**：将 `gs1_engine.py` 的核心逻辑镜像复制到前端 `lib/gs1.ts`
- **理由**：
  - 前端需要在无网络请求的情况下同步构建 HRI 与 GS1 Element String
  - 后端仍保留完整实现（数据入库前最终权威校验）
  - 两者保持同步修改约定（见下方"维护约定"）

### 6) 应用启动：lifespan 替代 on_event

**变更：`backend/app/main.py`**

```python

@asynccontextmanager
async def lifespan(app: FastAPI):
```

## 兼容性与约定（v2.0 起生效）

| 约定 | 内容 |
|------|------|
| **GS1 逻辑同构** | `lib/gs1.ts` 与 `gs1_engine.py` 须保持同步修改，任一处变更需同步更新另一处并写测试 |
| **条码渲染位置** | 所有预览均由 bwip-js 在客户端完成；后端不保存条码图像 |
| **保存时机** | 仅用户主动点击导出按钮时触发 `POST /generate`（不再每次生成都保存） |
| **历史预览** | 历史条目预览从 `LabelHistoryItem` 元数据在前端重渲染，无需额外接口 |
| **数据库迁移** | 新增表/列/索引须通过 Alembic 迁移文件操作，不可直接修改 `models.py` 后重启 |
| **日期格式** | 传输格式延续 `YY/MM/DD`（前端与后端 GS1 解析均已适配） |

## 核心调用链（v2.0 更新版）

### A. 生成 + 预览链路（无网络）

```
用户填写 DI/PI 表单
  → LabelForm.onSubmit()
  → useLabels.handlePreviewLocally(formData)    [同步]
      → lib/gs1.ts buildHri() + buildGs1ElementString()
      → setPreviewSource({ kind: "local", data: LocalPreviewData })
  → PreviewDialog 弹出
  → useBwipPreview(hri)                          [同步, bwip-js]
      → createDataMatrixSvg(hri)
      → createNormalizedGs1Svg(gs1ElementString)
  → PreviewTemplateCanvas 渲染 SVG
```

### B. 导出（唯一触发后端保存的时机）

```
用户点击"导出 PNG/SVG/PDF"
  → PreviewDialog.handleExport()
  → if (previewSource.kind === "local")
      → saveLabelToBackend(data, userId)         [POST /api/v1/labels/generate]
      → onSaved?.() → useLabelHistory.invalidateHistory()
  → exportPreviewNode(format)
```

### C. 历史查询链路（带缓存）

```
进入页面 / 翻页 / 筛选
  → useLabelHistory useQuery
      queryKey: ["label-history", userId, page, gtin, batchNo]
      → 若缓存未过期 (staleTime 30s)：直接返回缓存数据
      → 否则：GET /api/v1/labels/history?...
  → DataTable 渲染
  → 翻页时 keepPreviousData：不闪白屏
```

### D. 历史预览链路（即时，无网络）

```
用户点击历史条目"查看"
  → setPreviewSource({ kind: "history", data: LabelHistoryItem })  [同步]
  → PreviewDialog 弹出
  → useBwipPreview(item.hri)    [同步, bwip-js]
  → 立即渲染，无 loading
```
